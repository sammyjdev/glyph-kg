# GLYPH benchmark — graph vs vector vs hybrid

> **Pending first run.** This file is generated by `make benchmark` (or
> `scripts/run_benchmark.py`). No numbers are claimed until a real run with
> `GROQ_API_KEY` + `ANTHROPIC_API_KEY` produces them and commits
> `eval/benchmark-baseline.json` alongside this table.

## What the run measures

- **Arms:** graph-aware retrieval, a fair vector baseline, and their hybrid (ADR-G3),
  over the frozen query set in `eval/queries.json` (n=25).
- **Metrics:** GNOMON v1 — `faithfulness` and `context_precision` — each as a mean with
  a 95% percentile-bootstrap CI (seeded, via GNOMON `aggregate_metric`). Methodology in
  [ADR-G4](docs/decisions/dec-g4-eval-methodology.md).
- **Also reported:** real generation token counts, USD cost (Haiku 4.5 generation rates;
  judge tokens excluded), and mean latency per arm.

## How to produce the numbers

```bash
pip install -e ".[document,retrieval,embeddings,eval]"
export ANTHROPIC_API_KEY=...   # answer generation (Claude Haiku 4.5)
export GROQ_API_KEY=...         # OSS judge (default llama-3.3-70b-versatile)
make benchmark BOOK="<path to the Monster Manual PDF>"
```

This rewrites `eval/benchmark-baseline.json` (the committed regression baseline) and this
`METRICS.md`. Re-running `make benchmark-check BOOK=...` fails if a fresh run drifts beyond
tolerance (default 0.05) from the committed means.

## Honesty notes (P3.5)

- The relevance oracle in the query set is **KG-derived, not verified gold** — it inherits
  extraction errors (e.g. `ankheg → resists ácido`), kept on purpose so the comparison exposes
  graph noise.
- Only the two v1 metrics are reported; answer-relevance and context-recall are not built.
- The OSS judge is non-deterministic within measured variance; the CI width reflects that plus
  case-to-case variance.
